# Plan 02-04: End-to-end auth integration tests

## Objective
Prove each AUTH success criterion with an automated test driving an in-process
server + client. Includes the live ssh-agent Ed25519 happy path (ephemeral
agent), with an in-process-signer fallback when the agent is unavailable.

## must_haves
- truths:
  - "AUTH-01: a client key absent from authorized_keys is rejected at the handshake (no echo)"
  - "AUTH-02: known_hosts mismatch aborts the client; first contact writes TOFU and proceeds"
  - "AUTH-03: a forged CertificateVerify is rejected (signature not stubbed)"
  - "AUTH-04: the happy path signs via ssh-agent (live) or in-process fallback, exercising the same Signer code path"
  - "AUTH-05: flooding half-open connections stays bounded; un-authed conns close on timeout"

## Tasks

<task id="1" type="test">
<action>
Create crates/nosh-client/tests/auth.rs. Add a helper that mints a server host key + client
identity as in-process Ed25519 keys, writes a temp authorized_keys (client pub) and temp
known_hosts, starts an in-process server (bound to 127.0.0.1:0) and client using the
in-process-signer constructors, and runs a stream echo. Assert the handshake + echo succeed
(this is the AUTH-03/AUTH-04 in-process happy path). Use `tempfile` for the trust files.
</action>

<task id="2" type="test">
<action>
Add `unknown_client_key_rejected` (AUTH-01): authorized_keys does NOT contain the client key;
assert `connect`/first stream fails (handshake rejected), and the server logs no "connection
accepted" for it. Add `host_key_mismatch_aborts` (AUTH-02): pre-seed known_hosts with a WRONG
host key for the host; assert the client connect fails with a host-key-mismatch error. Add
`tofu_first_contact_records` (AUTH-02): empty known_hosts; assert connect succeeds and the
known_hosts file now contains the server key.
</action>
<read_first>

- crates/nosh-auth/src/verifier.rs
</read_first>
<acceptance_criteria>

</task>

<task id="3" type="test">
<action>
Add `forged_certificate_verify_rejected` (AUTH-03): construct a client whose presented cert
SPKI is the authorized client key but whose `AgentSigningKey`/`Signer` returns a signature from
a DIFFERENT Ed25519 key (a "forging" signer). Assert the handshake is rejected by the server's
real `verify_client_cert`+`verify_tls13_signature` path (proving signature verification is not
stubbed). This is the load-bearing negative test for PITFALL 5.
</action>
<read_first>
- crates/nosh-client/tests/auth.rs
- crates/nosh-auth/src/signer.rs
</read_first>
<acceptance_criteria>
- test `forged_certificate_verify_rejected` exists and passes (handshake fails)
</acceptance_criteria>
</task>

<task id="4" type="test">
<action>
Add `agent_ed25519_handshake_live` (AUTH-04), `#[ignore]`-gated unless `ssh-agent` and
`ssh-keygen` are on PATH. When present: spawn `ssh-agent -a <tmp.sock>`, `ssh-keygen -t ed25519`,
`ssh-add` the key, put its pubkey in authorized_keys, point the client `AgentSigner` at the
socket, and assert the full mutual-auth handshake + echo succeed. Kill the agent at the end.
If the tools are absent, the test no-ops with a skip log. Document this as the live-agent
verification path.
</action>
<read_first>

- crates/nosh-auth/src/signer.rs
</read_first>
<acceptance_criteria>

</acceptance_criteria>
</task>
