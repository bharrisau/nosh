<objective>
Wire the reattach protocol into the server: dispatch the first frame, emit the initial token on fresh opens, and on `Reattach` authorize via the registry (two-factor), rebind the pump onto the EXISTING orphaned slot, replay un-acked output, rotate+return the token in `ReattachOk`, and trim the buffer on inbound `Ack`. This delivers the server half of ROAM-02 + IDENT-02. Build on plan 06-02's registry methods — do not duplicate them.
</objective>

<context>
Current flow (read it): `handle_connection` resolves the connection, extracts `peer_identity` (server.rs:208), accepts ONE bidi stream, then `run_session` reads the first frame and requires `SessionOpen`. The output pump feeds `slot.push_output(&data)` then sends `PtyData`; the inbound arm handles `PtyData`/`Resize`/`SessionClose`. The `TransportLost` arm orphans the slot and spawns a watcher that awaits the orphaned shell's exit then `remove_slot`s by `Arc::ptr_eq`.

CRITICAL (the load-bearing safety point): on reattach the server must resume the SAME `Arc<SessionSlot>` instance returned by `registry.reattach` — NOT build a new one. The orphan-time watcher is still alive and `Arc::ptr_eq`-bound to that slot; `remove_slot` is idempotent, so the watcher remains the durable shell-exit observer across the reattach. Do NOT abort or replace that watcher. The reattach session's OWN end-handling (shell exit / new transport loss / client close) reuses the same teardown/orphan logic as a fresh session, with the same Arc.

Reattach is 1-RTT after the TLS handshake: the client sends `Reattach` as the first frame on the new connection's bidi stream; the server replies `ReattachOk` (or `ReattachErr`) on that same stream.
</context>

<task name="Implement run_reattach_session: authorize, rebind, replay, ack">
<read_first>
- crates/nosh-server/src/server.rs (run_session: the pump loop, output buffer feed, SessionEnd handling, the TransportLost watcher; PTY_CHUNK; nosh_proto::write_message)
- crates/nosh-server/src/registry.rs (SessionRegistry::reattach, SessionSlot::{replay_from, trim_acked, rotate_token, mark_active, token}, session lock for resize)
</read_first>
<action>
Add `async fn run_reattach_session(conn, peer, identity: NoshPublicKey, mut send, mut recv, last_acked_seq: u64, token: [u8;16], registry: Arc<SessionRegistry>) -> anyhow::Result<()>`:

1. Authorize: `match registry.reattach(&token, &identity)`:
   - `Err(_)` (ANY cause) → send `Message::ReattachErr` (no fields), `conn.close(CLOSE_PROTOCOL, b"reattach rejected")`, return Ok. Log `tracing::info!(identity = %identity.fingerprint(), "reattach rejected")` — NO token. ALL rejection causes take this identical path (uniform opaque error, D-07).
   - `Ok(slot)` → the slot is now `Reconnecting` (set inside reattach). Continue. The slot is the EXISTING orphaned Arc instance.
2. Compute replay: `let (chunks, replaying_from_seq, truncated) = slot.replay_from(last_acked_seq);`
3. Rotate the token: `let new_token = slot.rotate_token();` (single-use, D-05).
4. Send `Message::ReattachOk { new_token, replaying_from_seq, truncated }` on `send` FIRST (this is the 1-RTT reply). Then replay each chunk as `Message::PtyData { data }` in seq order (D-09: no dup/gap). If any send fails → treat as transport loss: re-orphan the slot (it's Reconnecting; transition back to Orphaned via `registry.orphan(&slot)` so it can be reattached again) and return. (Do not leave a slot stuck in Reconnecting on a failed rebind.)
5. Rebind the pump: rebuild the SAME output/input pump structure as `run_session` but operating on the existing `slot` (the PTY reader/writer were created at original open — they live with the Session inside the slot; the reader/writer handles were moved into the original pump tasks which died with the old connection). IMPORTANT: the PTY master reader/writer handles must be reachable for the new pump. If they were consumed by the original (now-dead) pump tasks, the slot must expose a way to re-acquire them. Read how `session::open` returns (reader, writer) and how `run_session` moved them into blocking tasks; design the slot so the reattach pump can read/write the live PTY. The simplest correct approach: have `SessionSlot` retain the ability to clone a fresh reader (`MasterPty::try_clone_reader`) and take/own a writer, OR store the reader/writer handles in the slot at open time so reattach can reuse them. Pick the approach that keeps the orphaned PTY's output buffered into `slot.push_output` continuously even while orphaned is NOT required (orphaned output accrues in the buffer only while a pump runs; per ARCHITECTURE the ring captures output while attached, and while orphaned the shell still runs and its output is read by... ) — RESOLVE THIS: while orphaned there is no pump reading the PTY, so output produced during the disconnect accumulates in the kernel PTY buffer and is read on reattach when the new pump's reader drains it. Ensure the reattach pump creates/reuses a reader that drains buffered-since-disconnect bytes, assigning them fresh seqs via `slot.push_output` as normal. (This means `replaying_from_seq` covers what was buffered BEFORE disconnect; post-disconnect kernel-buffered bytes get new seqs and flow normally once the pump resumes.)
6. After replay + pump rebind is established, transition the slot to Active: `slot.mark_active()`.
7. Run the pump loop identical to `run_session`'s (PtyData out with `slot.push_output`, inbound PtyData->PTY, Resize->slot.resize, SessionClose->ClientClosed) PLUS a new inbound arm: `Ok(Message::Ack { seq }) => { slot.touch(); slot.trim_acked(seq); }`. Reuse the same `SessionEnd` outcome handling (ShellExited/ClientClosed teardown via `registry.remove`/`remove_slot`; TransportLost re-orphans via `registry.orphan(&slot)`). Because the original orphan watcher is still alive and Arc::ptr_eq-bound, on a SECOND transport loss it (or a fresh watcher) will observe shell exit — do not leak. Prefer to NOT spawn a second exit watcher (the original is still valid); if you must (because the new pump owns a freshly-taken child), ensure `remove_slot` idempotency covers the overlap.

ALSO add the `Ack` inbound arm to the EXISTING fresh-session pump (`run_session`) so continuous acks work on the first connection too (D-08): `Ok(Message::Ack { seq }) => { slot.touch(); slot.trim_acked(seq); }`.

Keep all existing `tracing` spans/fields; add `identity = %identity.fingerprint()` to the reattach span. Never log `token`/`new_token`.
</action>
<acceptance_criteria>
- Reattach rejection sends fieldless `Message::ReattachErr` and closes; the same path serves every rejection cause (no oracle)
- Reattach success sends `ReattachOk { new_token, replaying_from_seq, truncated }` then replays `PtyData` for each chunk with seq > last_acked_seq in order
- The reattach resumes the SAME slot Arc (the one returned by `registry.reattach`); no new `SessionSlot::new` is called on the reattach path
- The slot is transitioned to `Active` after a successful rebind
- Both the reattach pump AND the fresh `run_session` pump handle inbound `Message::Ack { seq }` by calling `slot.trim_acked(seq)`
- No `tracing::` call references the token or new_token
- `cargo build -p nosh-server` succeeds
</acceptance_criteria>
</task>
