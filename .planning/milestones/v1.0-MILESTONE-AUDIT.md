## Summary

The v1.0 milestone is the M0–M2 architecture-validation spike: prove a single QUIC
connection on UDP/443 can carry a live interactive shell authenticated entirely from
the user's existing SSH-key identity, on Linux. All three phases — **QUIC Transport
Skeleton (P1)**, **SSH-Key Mutual Auth (P2)**, **PTY Session Core (P3)** — carry a
`status: passed` VERIFICATION.md, and the audit independently reproduced the gates:

- `cargo clippy --workspace --all-targets` — **PASS** (0 warnings)

All 21 v1 requirements (TRANS-01..05, AUTH-01..05, SESS-01..11) are satisfied. No
critical blockers, no unsatisfied or orphaned requirements. The remaining items are
intentional, documented M3+ seams and spike-scope limitations — not unplanned debt.

## Known-by-Design Limitations (recorded, not blockers)

1. **`Session.identity` is `None`** — the Phase-2 peer cert key is not yet threaded into the Phase-3 Session struct. This is the explicit M3 reattach wiring seam (`session::open(.., None)`), structured so M3 is additive rather than a refactor.
2. **Datagrams carry no session traffic** — only RFC 9221 enablement is proven (TRANS-03/04). Datagram-based predictive echo is M4.
3. **Single-account server, no privilege drop** (D-03) — shell runs as the server's account.
4. **Cold reattach not implemented** — `idle_since` + `Session` are the M3 seam only.
5. **Ed25519-only; RFC 7250 RPK deferred** — self-signed SPKI-pinning is the accepted first path per CONTEXT.
6. **SESS-03 / SESS-06 human-verified** — not headless-automatable; validated live at a real terminal and recorded PASSED.
