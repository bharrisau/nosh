<objective>
Add the client-side reconnect supervisor (D-01/D-02/D-08/D-10/D-11) and the integration tests that prove all four roadmap success criteria end-to-end against the server from plan 06-03. This closes ROAM-02 (1-RTT replay continuity) and IDENT-02 (two-factor, no-oracle) at the system level. Reuse the `persistence.rs` test harness — do not build a new server harness.
</objective>

<context>
Today `nosh-client/src/main.rs::run_interactive` opens one session over one connection and exits when the stream/connection ends. `client.rs` provides `connect`, `open_session`, `send_input`, `send_resize`, `run_session_collect`, `collect_until_close`. The headless drivers assume a `SessionOpen` first frame and ignore control frames — do NOT perturb them (Phase 5 tests depend on them); add NEW reattach-aware helpers.

The token lifecycle on the client: receive `SessionOpened { token }` right after the fresh `SessionOpen`; on each reconnect send `Reattach { token, last_acked_seq }` and replace the stored token with `ReattachOk.new_token`. The client tracks the highest output seq it has APPLIED to compute `last_acked_seq` (per the convention documented on `Message::Reattach` in plan 06-01). Because `PtyData` carries no seq and the QUIC stream is reliable+ordered, the client counts applied chunks from a known start seq: 0 on fresh open, `ReattachOk.replaying_from_seq` on reattach. Document this invariant.
</context>

<task name="Wire the reconnect supervisor into the client binary">
<read_first>
- crates/nosh-client/src/main.rs (run_interactive loop, RawModeGuard usage, connect/open_session flow, exit-code handling)
- crates/nosh-client/src/client.rs (connect, make_endpoint, the new reattach helpers)

<action>
Restructure `main.rs` so the interactive session runs under a reconnect supervisor (D-10/D-11). Hold across reconnects (in memory, D-01): `server_addr`, `host`, current `token: [u8;16]`, `highest_applied: u64` (or `Option<u64>` per the documented convention), and the `RawModeGuard` (created ONCE, restored once on final exit — do NOT drop/re-enter per attempt).

Flow:
1. Fresh connect: `connect`, `open_session_with_token` → store the initial `token`; set the seq baseline to 0 (nothing applied yet). Run the interactive pump (a refactor of `run_interactive`) that: renders `PtyData` and increments `highest_applied`; forwards keystrokes; sends `Resize` on SIGWINCH (existing debounce); periodically sends `Ack { highest_applied }` on a coarse cadence (e.g. a tokio interval every ~750ms, only when `highest_applied` advanced — D-08, Claude's-discretion cadence); and returns an outcome distinguishing CleanExit(code) (saw `SessionClose`) from TransportDrop (stream/conn error without SessionClose) from UserQuit (the explicit escape).
2. On CleanExit → exit with the code (existing behavior). On TransportDrop → enter the reconnect loop. On UserQuit → close cleanly and exit.
3. Reconnect loop (indefinite, D-10): print a minimal `\r\nnosh: reconnecting…\r\n` to stderr. Re-dial with exponential backoff capped (e.g. 250ms, x2 up to a 5–10s cap), retrying INDEFINITELY on dial/handshake failure (transport-level). On a successful dial: `connect` (re-runs TLS mutual handshake = factor 1), open a bidi stream, `send_reattach(token, highest_applied)`, `await_reattach_reply`:
   - `ReattachOutcome::Ok { new_token, replaying_from_seq, truncated }` → store `new_token`; set seq baseline so the next applied chunk continues from `replaying_from_seq`; if `truncated`, print a one-line `nosh: output truncated` stderr notice; resume the interactive pump. Reset backoff.
   - `ReattachOutcome::Err` → TERMINAL (D-11): the session is gone. Print `nosh: session ended` to stderr, break the loop, restore terminal (guard drop), exit non-zero. Do NOT keep retrying on `ReattachErr`.
4. Explicit quit path (D-11): pick an escape recognized ONLY while reconnecting (or a signal) that won't collide with normal shell input — e.g. Ctrl-C during the "reconnecting" window, or a documented `~.`-style chord checked only in the reconnect state. Document the chosen mechanism in a code comment and in the binary's `--help`/about text if practical. Ensure the `RawModeGuard` restores the terminal on this path.

Keep `collect_client_env`, the SIGWINCH debounce, and raw-mode semantics. The terminal must be restored exactly once on final exit regardless of how many reconnects occurred.
</action>
<acceptance_criteria>
- The client re-dials and sends `Reattach { token, last_acked_seq=highest_applied }` after a transport drop, applying replayed output to continue from `replaying_from_seq`
- The stored token is replaced by `ReattachOk.new_token` on each successful reattach
- Backoff is exponential and capped; retry is indefinite on transport-level failure; `ReattachErr` ends the loop (no infinite retry on a gone session)
- A minimal `reconnecting…` stderr notice is printed; a `truncated` notice is printed when the server signals truncation
- The client sends `Ack` on a coarse cadence (not per-chunk)
- An explicit quit mechanism exists and is documented in a code comment
- `cargo build -p nosh-client` succeeds
</acceptance_criteria>
</task>
