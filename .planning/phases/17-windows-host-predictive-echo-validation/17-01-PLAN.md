<environment_facts>
<!-- Verified by orchestrator + planner against the codebase. Do NOT re-derive these. -->

- We are running on the physical Windows host now (unblocks this phase).
- `nosh-client` builds clean on Windows:
  - debug:   `cargo build -p nosh-client`
  - release: `cargo build --release -p nosh-client --target x86_64-pc-windows-msvc`
    → binary at `target\x86_64-pc-windows-msvc\release\nosh-client.exe`
- `nosh-server` is **Linux-only**. The server MUST run on a separate, network-reachable Linux box.
- This phase REQUIRES, in addition to the Windows host:
  - a Linux machine running `nosh-server` (with the operator's Ed25519 public key in its
    `authorized_keys`),
  - a REAL network path between them (NOT loopback, NOT same-host) so SRTT > 0 and predictions
    actually engage,
  - a WireGuard tunnel the operator can stand up and tear down mid-session (for the migration test).

- Client CLI flags (verified in `crates/nosh-client/src/main.rs`):
  - `--addr <Ip>`           server IP (default 127.0.0.1)
  - `--port <u16>`          server port (default 4433)
  - `--host <name>`         QUIC SNI + known_hosts lookup key (default localhost)
  - `--identity-file <path>` on-disk Ed25519 private key (the ONLY Windows auth path;
                             defaults to `%USERPROFILE%\.ssh\id_ed25519` when omitted)
  - `--predict adaptive|always|never`  (default adaptive). Use **`--predict always`** for the
                             timing test to force predictions on regardless of measured RTT.
  - `--status`              surfaces measured SRTT in the terminal title (`nosh: <N>ms`).

- D-17-02a latency instrumentation ALREADY SHIPPED (do NOT plan to build it). At
  `crates/nosh-client/src/main.rs` ~836–856, on each confirmed prediction the client logs:
  `tracing::debug!(target: "nosh::predict", event="confirm", epoch, latency_ms, "prediction confirmed")`.
  It is surfaced through the `RUST_LOG` EnvFilter (tracing_subscriber fmt → stderr). Only timing +
  epoch are logged; never character content (T-15-08).

- **Latency capture mechanism (use EXACTLY this — D-17-02):** set
  `RUST_LOG=nosh::predict=debug` and redirect **stderr** to a file so the debug lines do not corrupt
  the interactive TUI on stdout. In PowerShell:
  `$env:RUST_LOG="nosh::predict=debug"; .\nosh-client.exe --addr <ip> --port <p> --host <h> --predict always 2> predict.log`
  After the session, `predict.log` contains `latency_ms=<n>` per confirmed prediction; the operator
  records representative numbers (e.g. min / median / max) in the doc.
</environment_facts>
