<objective>
Add observability for the roaming feature that live validation confirmed works but logs nothing. When a session's QUIC peer address changes (connection migration — e.g. Wi-Fi→cellular), the server emits an INFO log with session_id and old→new address. quinn 0.11 has no migration callback, so detect by polling `connection.remote_address()` inside the existing session loop and logging on change.

Purpose: ROAM-01 migration was observed working from a native Windows client but the server is silent about it; OBS-01 closes that observability gap.
Output: A periodic remote-address poll in the server's session select loop that logs migration events at INFO.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| migrating peer → server session | A peer address change is attacker-influenceable; the log must be observational only and must not weaken the connection's existing QUIC path-validation guarantees |
| log output → operator | Logged addresses are operational metadata, not secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-07 | Spoofing | migration log trusting remote_address | accept | The log reflects quinn's already-path-validated remote_address; nosh adds no authorization decision here — it only reports. QUIC connection-ID routing + path validation remain the security control; this change cannot be used to bypass them |
| T-09-08 | Denial of Service | log spam on rapid addr flapping | mitigate | Poll cadence (500ms) bounds log frequency to at most ~2/s per session, and a log only fires on an actual change; no per-packet logging |
| T-09-09 | Information Disclosure | logging peer IP:port | accept | Peer address is standard operational/observability data (already present in the session span as %peer); INFO level is appropriate, no secret material logged |
</threat_model>
