# Phase 999.3: Client terminal-rendering correctness pack (platform-agnostic) - Context

**Gathered:** 2026-06-05
**Status:** Ready for planning
**Mode:** discuss (interactive, via /gsd:autonomous --interactive — promoted from backlog)

<domain>
## Phase Boundary

Resolve the terminal-handling defects surfaced during Phase 17 live validation. All items
reproduce on a Linux client — fix and test on Linux where the full test suite compiles.

In scope (5 items): BUG-H (clear-on-connect / blank-cell painting / Ctrl-L), BUG-E (backspace
past prompt), BUG-F (Enter after `read -s` noecho), the typematic / fast-typing glitch, and
the D-17-02a latency-instrumentation rework (folded in per user decision).

Out of scope: anything not flagged during Phase 17 validation; new prediction capabilities;
the already-fixed Windows-specific BUG-G (ConPTY startup size-sync lag — shipped in Phase 17).
This is a correctness/quality pack, not a feature phase.
</domain>

<decisions>
## Implementation Decisions

### BUG-E — backspace/left past prompt start
- **D-01:** The predictor learns its clamp floor by recording the **confirmed cursor column at
  the start of each fresh input epoch**. `PredictBackspace` and `PredictCursorLeft`
  (`predictor.rs:330-346`) clamp at that epoch-start column instead of col 0. No prompt-string
  parsing — the floor is simply where the user's typing began after the prompt. Mirrors Mosh's
  overlay model. Must not regress the existing CR-01 `sync_cursor_from_confirmed` behavior.

### Typematic / fast-typing glitch
- **D-02:** Replace the flat `>4-byte → BulkSuppressed` rule (`predictor.rs:625-628`) with
  **content inspection**: if a stdin batch is entirely printable single-width characters
  (typical key-repeat / fast typing), predict each character in sequence. Suppress only on
  genuinely un-modelable content (escape sequences, control bytes, or mixed batches).
  Bracketed-paste suppression (`BracketedPasteStart`/`End`) MUST remain intact. CJK / wide-char
  handling (the existing `>4 bytes` test for '你') must still pass.

### BUG-H — clear-on-connect / blank cells / Ctrl-L
- **D-03:** Fix where the bug actually is, evidence-driven. Client side (`screen.rs`):
  (1) emit a physical clear to the terminal on connect so prior content doesn't bleed through,
  and (2) paint blank cells as spaces in the full-framebuffer diff so erased regions are
  actually cleared. Server side (`nosh-server/src/terminal.rs` ED/clear handling): change ONLY
  if a reproduction test proves the server's StateDiff for Ctrl-L / ED is wrong. No speculative
  server-terminal-model edits.

### D-17-02a — latency instrumentation
- **D-04:** **Included in this pack** (user decision). Rework the latency hook in `run_pump` so
  it measures **per-keystroke RTT** (predicted→confirmed for an individual keystroke), not
  epoch-confirmation time (which includes user think-time). Goal: timing evidence fine enough to
  back the "never worse than no prediction" claim. Keep it behind the existing instrumentation
  path; do not change user-visible behavior.

### BUG-F — Enter after `read -s` noecho prompt doesn't advance the line
- **D-05:** (locked to recommended — not separately discussed) When the noecho epoch ends, the
  predicted caret may be stale. Fix: on noecho-epoch end / `EpochReset`, **sync the predicted
  caret from the confirmed StateDiff cursor** rather than relying on the stale predicted caret,
  so the post-`read -s` Enter advances the line correctly. Investigate the
  `EpochReset`/`become_tentative`/render path (`predictor.rs:308-374, 576-604`).

### Test strategy
- **D-06:** (locked to recommended) Every bug gets a **deterministic Linux reproduction test
  that fails before the fix and passes after** (adversarial — written to reproduce the reported
  symptom, not to rubber-stamp). Prefer unit tests against `predictor.rs` / `screen.rs` where
  the defect is pure client logic; use an integration/PTY-level test only where a unit test
  can't reach (e.g. server ED handling if D-03 touches the server). The full client test suite
  must compile and pass on Linux.

### Claude's Discretion
- Exact internal representation of the epoch-start clamp floor (field on the overlay vs derived).
- Whether content inspection for D-02 lives in `classify_input` or a new batch-classifier helper.
- Physical-clear escape sequence choice on connect (e.g. ED2 + cursor-home vs full reset) so
  long as it does not corrupt the confirmed/physical grid invariant.
- Per-keystroke RTT storage shape for D-04 (extend the existing HashMap hook in `run_pump`).
</decisions>

<specifics>
## Specific Ideas

- These are real daily-driver bugs found by the maintainer during live Windows-vs-Linux use on
  2026-06-02; the triage already proved each reproduces on a Linux client (no `#[cfg(windows)]`
  in the faulty paths). Trust the triage's root-cause file/line pointers as the starting map.
- Invariant to preserve: all output to the local terminal goes through
  `ClientScreen.render_to_stdout()` — never direct `stdout.write_all` (v1.2 roadmap decision).
- Invariant to preserve: `predicted_cursor` snapping (CR-01) — datagram confirmation syncs the
  predicted caret only when pending is empty; D-01/D-05 must not break left/right speculative
  motion that diverges the predicted caret without enqueuing a pending cell.
</specifics>
