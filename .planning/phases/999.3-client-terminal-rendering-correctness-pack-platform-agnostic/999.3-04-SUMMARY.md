# Phase 999.3 Plan 04: D-03b connect-clear wiring + D-04 per-keystroke RTT Summary

Two `run_pump` changes in `main.rs`: wire `emit_connect_clear` on connect (D-03b, BUG-H fix completion) and rework the latency hook from epoch-grained to per-keystroke RTT instrumentation (D-04).

## What Was Built

**Task 1 — D-03b connect-clear wiring:**

Immediately after `let mut screen = nosh_client::screen::ClientScreen::new(cols, rows);` in `run_pump`, the code now calls `screen.emit_connect_clear(&mut init_buf)` (the method added in plan 999.3-03), buffers the result into a `Vec<u8>`, then `stdout.write_all(&init_buf).await` + `stdout.flush().await`. This fires once on connect before any datagram arrives, clears the physical terminal of prior content (BUG-H), and resets the physical model so the first `render_with_predictor` performs a full repaint. A `render_to_stdout` is NOT called here (confirmed grid is empty at connect — correct per the plan). Errors are `tracing::warn!`-logged with `reset_physical()` called as fallback. Comment tags D-03 / BUG-H.

**Task 2 — D-04 per-keystroke RTT rework:**

The existing D-17-02a hook used `HashMap<u64 epoch, Instant>` and measured epoch-confirmation latency — which included user think-time (multiple keystrokes could accumulate in one epoch window). D-04 reworks this to true per-keystroke RTT:

- Map key changed from `prediction_epoch (u64)` to `keystroke_id (u64)` — a monotonic counter incremented once per forwarded keystroke batch. Type signature is still `HashMap<u64, Instant>`; same variable name `predict_enqueue_times` at the same declaration location.
- Enqueue side: `keystroke_id += 1; predict_enqueue_times.insert(keystroke_id, Instant::now());` — records a fresh timestamp for every keystroke batch, logged with `keystroke_id` and `epoch_required` fields.
- Confirm side: when `epoch_after_cull > epoch_before_cull` (a prediction epoch advanced), a `confirmed_watermark = keystroke_id` snapshot is taken, all entries with id <= watermark are collected, sorted, drained, and logged individually with `keystroke_id`, `epoch`, and `latency_ms` fields (per-keystroke RTT). The `retain(|&k, _| k > confirmed_watermark)` pruning is preserved to prevent unbounded map growth (D-04 invariant).
- Keystroke send path (`client::send_input` on reliable stream) is completely unchanged — this is instrumentation only.

D-04 is intentionally instrumentation-only with no new behavioral test: there is no observable behavior change (the timing logs are at `tracing::debug!` level and carry no character content). Evidence of correctness is the full test suite passing with the send path unmodified.
