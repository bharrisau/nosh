# Phase 999.3: Code Review Report

**Reviewed:** 2026-06-05
**Depth:** standard

## Summary

Three files were reviewed covering the D-01 through D-05 correctness fixes and the D-04 per-keystroke RTT instrumentation. The mechanical implementation of BUG-E (backspace/cursor-left clamping at `epoch_start_col`), BUG-H (connect clear and blank-cell emission), and D-02 (PredictBatch typematic fix) are all sound. The cull index-safety and D-04 drain/prune logic are correct in the common case. One critical defect was found in the D-05/D-01 interaction that leaves `epoch_start_col` stale after every `EpochReset` (Enter), undermining the BUG-E prompt-boundary clamp for the session's next input epoch. Three warnings were also identified.

## Critical Issues

### CR-01: `reset_with_cursor` sets `epoch_start_col` to the pre-Enter cursor position and blocks the subsequent `sync_cursor_from_confirmed` from correcting it

**File:** `crates/nosh-client/src/predictor.rs:602-613`

**Issue:**

`reset_with_cursor` (the D-05 fix for BUG-F) assigns `self.epoch_start_col = confirmed.col` and then sets `self.needs_epoch_start_sync = false`. The `confirmed` argument is `screen.confirmed_cursor()` at the moment the user presses Enter — which is the cursor position *before the server has processed the Enter*. It reflects wherever the cursor was in the previous typing epoch (e.g. after the last echoed character).

After the Enter the server advances to a new prompt line. The next `sync_cursor_from_confirmed` call (in the datagram arm after the first post-Enter diff) correctly updates `predicted_cursor` to the new prompt position, but it *cannot* update `epoch_start_col` because `needs_epoch_start_sync` is `false`. The floor is therefore stuck at the stale pre-Enter column for the *entire* next input epoch.

Concrete symptom: suppose the prompt is `user> ` (epoch_start_col should be 6 for the next epoch). The Enter was pressed when the cursor was at col 9 (after typing "abc"). `reset_with_cursor` sets `epoch_start_col = 9`. For the next epoch the backspace and cursor-left clamps stop at col 9, not col 6, so the user cannot backspace all the way back to the start of their new input.

In the noecho case (the primary BUG-F scenario): `read -s` prompt "Password: " leaves the confirmed cursor at col 10 (end of prompt, since nothing was echoed). Enter fires `reset_with_cursor({col: 10})`. `epoch_start_col = 10`. The next prompt may be at col 0 on a new row, but the floor is now col 10 — backspace stops nine columns too early.

This flaw means the D-01/BUG-E fix is silently bypassed after every Enter that goes through `EpochReset`. The BUG-E regression tests only call `reset()` directly and `sync_cursor_from_confirmed` with a crafted `make_diff_with_char` that puts confirmed_cursor at the expected col; they do not exercise the Enter → `reset_with_cursor` → next datagram → `sync_cursor_from_confirmed` chain that is the real call path from `run_pump`.

**Fix:**

`reset_with_cursor` should force-sync `predicted_cursor` (the BUG-F intent) but then re-arm `needs_epoch_start_sync` so the next `sync_cursor_from_confirmed` call — after the server confirms the new prompt position — captures the real epoch-start column:

```rust
pub fn reset_with_cursor(&mut self, confirmed: CursorPos) {
    self.pending.clear();
    self.become_tentative();
    self.cursor_motion_pending = false;
    // D-05 (BUG-F): force-sync predicted caret directly (bypasses CR-01 guard;
    // pending is empty so the guard's purpose does not apply).
    self.predicted_cursor = confirmed;
    // D-01: re-arm needs_epoch_start_sync so the NEXT sync_cursor_from_confirmed
    // (after the server confirms the new prompt position) captures the correct
    // epoch_start_col. Do NOT use confirmed.col here — that is the pre-Enter
    // position, not the new prompt boundary.
    self.needs_epoch_start_sync = true;
}
```

Remove the `self.epoch_start_col = confirmed.col;` line. The `needs_epoch_start_sync = true` path in `sync_cursor_from_confirmed` will capture the correct column once the first post-Enter datagram arrives.

## Warnings

### WR-01: `PredictLineStart` (Home / Ctrl-A) ignores `epoch_start_col` — inconsistent with BUG-E fix

**File:** `crates/nosh-client/src/predictor.rs:398-401`

**Issue:**

`PredictCursorLeft` and `PredictBackspace` both clamp at `self.epoch_start_col` (the BUG-E fix). `PredictLineStart` — triggered by Home, Ctrl-A, CSI H, CSI 1~, SS3 H — unconditionally sets `self.predicted_cursor.col = 0`. This places the predicted caret *before the prompt* whenever the prompt starts at a non-zero column (e.g. `user> ` starts at col 0 but the input area starts at col 6; Home in a readline-based shell moves to the start of the *input*, which is col 6, not col 0).

The visible effect: after pressing Home, the predicted caret jumps to col 0 (overlapping prompt text), while the server places the cursor at col 6 (readline's line-start). The next cull sees a mismatch, which resets the epoch — a brief visual glitch each time Home is pressed. With `cursor_motion_pending = true` the snap-back is also blocked until the reset fires, so the wrong-position caret is held for a full RTT.

This is the same class of defect that BUG-E fixed for backspace and cursor-left, and was not extended to `PredictLineStart`.

**Fix:**

```rust
InputAction::PredictLineStart => {
    // Clamp at epoch_start_col (the prompt boundary) — same fix as BUG-E for
    // PredictBackspace and PredictCursorLeft. Home/Ctrl-A moves to the start
    // of the current input, not to column 0.
    self.predicted_cursor.col = self.epoch_start_col;
    self.cursor_motion_pending = true;
}
```

### WR-02: `predict_enqueue_times` grows without bound during extended noecho / always-reset sessions

**File:** `crates/nosh-client/src/main.rs:850-854, 1025-1052`

**Issue:**

The D-04 RTT map (`predict_enqueue_times`) is pruned only inside the `if epoch_after_cull > epoch_before_cull` block (lines 1025-1052). This block executes only when `cull()` advances `confirmed_epoch`, which requires at least one `Validity::Correct` prediction confirmation.

During any extended noecho session (`stty -echo`, `read -s`, `sudo` password prompt) or after repeated `EpochReset` events (tab-completion storms, paste-start, arbitrary ESC sequences), `confirmed_epoch` never advances. Each forwarded keystroke still increments `keystroke_id` and inserts into `predict_enqueue_times` (line 1191), but the prune block never runs. The map grows without bound for the duration of the noecho session.

At interactive typing speeds (~8 chars/sec) a 60-second password entry adds ~500 entries (~12 KB). For an automated pipe that sends characters with no echo, the growth is faster. Over a long-running session with many noecho prompts the map accumulates entries from each without inter-prune cleanup.

**Fix:**

Add a bounded fallback prune triggered by map size, or prune on every `EpochReset` / `BulkSuppressed` event (since those represent a clean break in the keystroke-to-epoch linkage):

```rust
// In the stdin arm, after predictor.on_input():
// If the overlay reset (EpochReset or BulkSuppressed), all in-flight keystrokes
// will never be confirmed. Prune entries up to the current watermark.
if predictor.pending_len() == 0 {
    // Approximate: prediction reset — prune stale entries.
    let watermark = keystroke_id;
    predict_enqueue_times.retain(|&k, _| k > watermark);
}
```

Alternatively, cap the map at a maximum size (e.g. 1000 entries) and evict oldest entries if exceeded, trading precision for safety.

### WR-03: OSC 0/2 title from server is not sanitized before re-emission to the local terminal

**File:** `crates/nosh-client/src/main.rs:965-975`

**Issue:**

The OSC 52 clipboard path (lines 953-962) strips ESC (`\x1b`) and BEL (`\x07`) from both `sel` and `b64` before interpolating them into the escape sequence. The OSC 0/2 title path does not apply any equivalent sanitisation:

```rust
let osc02 = format!("\x1b]0;{title}\x07");
```

A server-controlled `title` string containing an embedded BEL (`\x07`) would terminate the OSC sequence early, allowing any bytes after the BEL to be interpreted by the local terminal as raw escape sequences. A title containing ESC could introduce an entirely new escape sequence. While the data path is authenticated (the server must pass the client's SSH-key auth), a compromised server or a vulnerability in the server's OSC parser could exploit this to inject terminal sequences into the client terminal.

The WR-01 comment on the clipboard path ("defense-in-depth") explicitly acknowledges this concern; the same logic applies to the title path and the fix was not applied consistently.

**Fix:**

Apply the same filter to `title` before interpolation:

```rust
TerminalControlPayload::Title { title } => {
    if !status {
        let clean_title: String = title
            .chars()
            .filter(|&c| c != '\x07' && c != '\x1b')
            .collect();
        let osc02 = format!("\x1b]0;{clean_title}\x07");
        let _ = stdout.write_all(osc02.as_bytes()).await;
        let _ = stdout.flush().await;
    }
}
```

### IN-01: `emit_connect_clear` error path in `run_pump` does not call `reset_physical()` as a fallback

**File:** `crates/nosh-client/src/main.rs:798-800`

**Issue:**

When `screen.emit_connect_clear(&mut init_buf)` returns `Err`, only a warning is logged (lines 798-800). `reset_physical()` is NOT called on this error path, even though `emit_connect_clear`'s internal implementation calls `reset_physical()` only on success (after the write). If a future refactor changed the writer type from `Vec<u8>` to one that could fail (e.g. a real file or socket), the physical grid would remain unclean and the first render would be a partial repaint instead of a full one.

In practice this is unreachable today — `Vec<u8>::write_all` is infallible — but the asymmetry with the write-failure paths at lines 803-807 (which do call `reset_physical()`) creates a misleading code pattern.

**Fix:**

Add `screen.reset_physical()` to the `emit_connect_clear` error branch for consistency:

```rust
if let Err(e) = screen.emit_connect_clear(&mut init_buf) {
    tracing::warn!("emit_connect_clear error: {e}");
    screen.reset_physical(); // ensure a full repaint even if the write failed
}
```
