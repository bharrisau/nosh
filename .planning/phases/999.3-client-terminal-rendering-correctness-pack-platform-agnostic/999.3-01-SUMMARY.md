# Phase 999.3 Plan 01: D-01 BUG-E epoch-start clamp + D-05 BUG-F EpochReset cursor sync — Summary

**One-liner:** `epoch_start_col` clamp floor gates backspace/left at the prompt boundary (D-01) and `reset_with_cursor` force-syncs the predicted caret on `EpochReset` so the post-`read -s` Enter advances the line (D-05).

## What Was Built

Two predictor correctness fixes in `crates/nosh-client/src/predictor.rs`, each with a TDD RED/GREEN cycle and adversarial fail-before/pass-after tests (D-06).

### Task 1: D-01 BUG-E — epoch-start clamp floor

Added `epoch_start_col: u16` field to `PredictionOverlay` (initialized to 0) and a companion `needs_epoch_start_sync: bool` flag. The flag is set by `reset()` and cleared by the first `sync_cursor_from_confirmed` in the new epoch that captures `epoch_start_col = confirmed.col`. This ensures that mid-epoch datagram confirmations (server echoing back typed chars, emptying `pending`) do NOT overwrite the prompt-boundary floor.

Changed `PredictBackspace` guard from `> 0` to `> self.epoch_start_col` and `PredictCursorLeft` guard likewise. Inner bodies are unchanged.

### Task 2: D-05 BUG-F — EpochReset cursor sync

Added `reset_with_cursor(confirmed: CursorPos)` method that performs the same body as `reset()` plus direct assignment of `self.predicted_cursor = confirmed` and `self.epoch_start_col = confirmed.col`. The direct assignment intentionally bypasses the CR-01 guard (pending is being cleared, so the guard's purpose does not apply) — documented with a doc-comment referencing D-05 and CR-01.

Split the `EpochReset | BulkSuppressed` match arm in `on_input` into two separate arms:
- `EpochReset` calls `self.reset_with_cursor(screen.confirmed_cursor())`
- `BulkSuppressed` keeps calling `self.reset()` (no confirmed cursor available at bulk-input time)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] epoch_start_col flag needed to prevent mid-epoch overwrites**

- **Found during:** Task 1 GREEN — `bug_d_arrow_motion_moves_caret_and_survives_sync` failed because `sync_cursor_from_confirmed` ran after server confirmed "hello" (pending empty, cursor at col 5) and set `epoch_start_col = 5`, then cursor-left from col 5 was clamped.
- **Issue:** Setting `epoch_start_col` on every `sync_cursor_from_confirmed` call conflated "prompt boundary after reset" (correct) with "cursor after server echo confirmation" (wrong — not a fresh epoch start).
- **Fix:** Added `needs_epoch_start_sync: bool` flag, set by `reset()`, cleared after first capture. Means `epoch_start_col` is updated only once per epoch, at the true fresh-epoch start. Updated BUG-E tests to call `overlay.reset()` before `sync_cursor_from_confirmed` to simulate the real Enter → new-prompt sequence.
